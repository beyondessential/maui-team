# Glossary

Short definitions of terms used across Maui repos. The architecture doc and
external standards are authoritative; this is a lookup table.

## Systems

- **Tamanu** — open-source EHR (BES). Source of clinical data.
- **Tupaia** — open-source visualisation platform (BES).
- **mSupply** — health-commodity management system.
- **SENAITE** — lab information system.
- **FluTracking** — ILI surveillance data.
- **NIWA** — NZ weather/climate data source.
- **FSM EHR** — predecessor EHR migrated from in `fsm-data-migration`.

## Maui repos

- **`maui-team`** — this repo. Imported by every other repo as `.maui` submodule.
- **`tamanu-source-dbt`** — canonical dbt project for Tamanu reporting.
- **`tamanu-dbt-<deployment>`** — per-deployment overrides of `tamanu-source-dbt`.
- **`data-staging`** — external dbt package being merged into `tamanu-source-dbt`.
- **`data-lake`** — Dagster orchestration. Rename to `bes-data-pipelines` in flight.
- **`fsm-data-migration`** — one-off FSM → Tamanu migration.
- **`datatools`** — CLI for Tamanu/Tupaia data tasks.

## Three things named `data-lake`

- **`data-lake` RDS** *(retired)* → replaced by `data-warehouse` RDS.
- **`data-lake` repo** *(rename in flight)* → becoming `bes-data-pipelines`.
- **Tupaia `data-lake` service** *(unrelated)* — Tupaia-side data-broker being
  unwound (D9). Name collision is coincidental.

## Standards bodies

- **OMOP** — clinical analytics data model, governed by OHDSI. D2 layer mapping.
- **OMOP-lite** — BES subset of OMOP used in `can__` / `der__`.
- **OHDSI** — steward of OMOP.
- **Athena** — OHDSI concept lookup, <https://athena.ohdsi.org/>.
- **FHIR** — exchange standard (out of scope per D4).
- **WHO SMART / DAK**, **PEPFAR**, **DHIS2**, **MANA**, **SDG** — indicator
  catalogues referenced in `metric_definitions.csv`.

## dbt layer prefixes

Full table: [`standards/dbt-conventions.md`](standards/dbt-conventions.md).

- **`base__`** — only layer reading `public.*`. Filters deleted rows. D10.
- **`ref__`** — OMOP health-system wrappers (care site, location, provider).
- **`lkp__`** — curated lookups (age bands, mappings, codings). Seed-backed.
- **`surveys`** — generated pivot of survey responses (one column per question
  per form). Regenerated by script; never edited manually.
- **`can__`** — canonical OMOP-lite clinical events.
- **`der__`** — derived elements: cohorts, eras, episodes.
- **`metric__`** — health indicators. Mandatory `metric_definitions.csv` entry.
- **`int__`** — shared intermediate logic. Always `ephemeral`.
- **`ds__`** — consumer-shaped datasets joining `can__` / `der__` / `metric__`.
- **`models/reports/`** — Tamanu reports. `order by` allowed.

Legacy (in active use): **`fct__`**, **`dim__`** (→ `ref__` / `can__`).

## Output surfaces

- **Tamanu report** — production view, paired with JSON config, ships via
  compiled bundle.
- **Compiled bundle** — per-release SQL artefact in
  `tamanu-source-dbt/compiled/v<version>/`. Contains the transitive closure of
  every report.
- **Production-safe** — SQL valid as a view in production (no Python models, no
  replica-only columns, no heavy window functions).
- **Replica** — bounded-stale copy of production. DevOps-owned. Tupaia reads it.
- **Tupaia Data Table** — Tupaia primitive that reads one table/view. D9
  consumption surface for `metric__`.
- **External Database Connection** — Tupaia admin-panel construct wiring a
  Data Table to its source DB.
- **Line list** — Tamanu report shape: one row per patient/encounter.
- **Sensitive report** — variant including encounters from sensitive facilities.
  Naming: `sensitive-<name>.sql`.

## Workflow concepts

- **`reporting_*` target** — compiles the production bundle (forces `view`).
- **`analytics_*` target** — runs against the replica (`incremental` / `table`
  permitted).
- **Env-aware materialisation** — model picks materialisation via
  `target.name.startswith('reporting_')`.
- **Release deployment** — environment that generates the compiled bundle.
- **Clone deployment** — per-deployment env that validates a bundle before
  production promotion. Ops-owned.
- **Variant** — deployment-specific `metric__` override flagging meaningful
  divergence via the `variant_id` column.

## SDD

Full conventions: [`standards/sdd-conventions.md`](standards/sdd-conventions.md).

- **BL** — business logic clause (`BL-001`).
- **AC** — acceptance criterion.
- **OQ** — open question (owner + due).
- **DV** — divergence between current code and spec (Mode A only).
- **DQ** — data quality check (migration specs).
- **Spec anchor** — `-- BL-001:` / `# BL-001:` comment linking code to the
  spec clause.
