# dbt Model Metadata

All dbt models must include a `meta:` block in their YAML file. This file defines the valid values for each field.

## Usage

```yaml
meta:
  owner: "bes-maui"              # see Owner values
  domain: "clinical"         # see Domain values
  tier: "gold"               # see Tier values
  pii: true                  # true if model contains any PII columns
  classification: "restricted"   # see Classification values
```

Assign the highest applicable classification level when a model contains data from multiple levels.

---

## Owner values

| Owner | When to use |
|---|---|
| `bes-maui` | Models built and maintained by the Maui team (mSupply pipelines, dbt transformations) |
| `bes-tamanu` | Tamanu source models — raw data maintained by the Tamanu team |
| `flutracking` | FluTracking surveillance data |
| `niwa` | NIWA weather and climate data |

Add a new owner value when ingesting data from an additional external source.

## Domain values

Domains classify datasets by subject matter. They are propagated to DataHub for discovery and governance. Assign the domain that best describes **what the data is about**, not who owns it or who can access it.

| Domain | Description |
|---|---|
| `clinical` | Patient-level care data: encounters, diagnoses, medications, program registry, births, deaths |
| `admin` | Facility and operational management: billing, scheduling, user/role configuration |
| `supply_chain` | Health commodity management: stock levels, orders, dispensing, cold chain (mSupply) |
| `surveillance` | Population-level disease monitoring: influenza (FluTracking), dengue, outbreak tracking |
| `environmental_health` | Environmental risk and community health: food retailer inspections, household surveys |
| `asset_management` | Physical assets and equipment |
| `external` | Context data not generated by Maui systems: weather and climate data (NIWA) |

### Domain guide

- **`clinical`** — Does this describe something that happened to a patient? (encounter, test, prescription, diagnosis, birth, death) → `clinical`
- **`admin`** — Is this about how a facility or program is run rather than about patients? (billing, roles, config) → `admin`
- **`supply_chain`** — Does this track the movement or status of a physical commodity? (stock, order, shipment) → `supply_chain`
- **`surveillance`** — Is this population-level disease or outbreak monitoring data? → `surveillance`
- **`environmental_health`** — Is this about environmental risk factors or community conditions rather than individual patients or diseases? → `environmental_health`
- **`asset_management`** — Is this tracking a physical asset (equipment, vehicle, facility infrastructure)? → `asset_management`
- **`external`** — Is this data sourced from an external system that Maui does not generate or control, used as environmental or analytical context? → `external`

### Source-to-domain reference

| Source / repo | Typical domain |
|---|---|
| Tamanu — clinical tables (encounters, diagnoses, medications, program registry, births, deaths) | `clinical` |
| Tamanu — admin tables (billing, facilities, users, roles) | `admin` |
| mSupply | `supply_chain` |
| FluTracking | `surveillance` |
| Dengue surveillance | `surveillance` |
| NIWA weather/climate | `external` |
| Environmental health (food retailers, household surveys) | `environmental_health` |
| Asset management | `asset_management` |

## Tier values

Tiers indicate the quality and readiness of a dataset. They map onto dbt layers:

| Tier | dbt layers | Meaning |
|---|---|---|
| `bronze` | `sources`, `logs` | Raw, as-ingested. No transformation, no guarantees. Source-faithful. |
| `silver` | `bases`, `intermediate` | Cleaned and tested. Deleted records removed, types cast, PII masked. Trusted structure, not yet user-facing. |
| `gold` | `facts`, `surveys`, `datasets`, `reports` | Business-ready. Joined, aggregated, documented. Safe for analysts and Tupaia to consume. |

Edge cases:
- External sources (NIWA, FluTracking raw) → `bronze`, even if they arrive pre-cleaned, because Maui does not control or test them
- mSupply bases → `silver` once cleaned, same as Tamanu bases
- Cross-instance aggregations → `gold` if user-facing, `silver` if intermediate joins

## Classification values

Classification levels indicate the sensitivity of data and the handling controls required. They are defined in BES's [Data Classification Policy](https://beyond-essential.slab.com/posts/p-9-data-classification-policy-4e4cypfy).

| Level | Description |
|---|---|
| `restricted` | Highest sensitivity. Includes personal health information, PII, credentials, and data subject to regulatory or contractual obligations. Strict access controls and encryption required. |
| `confidential` | Sensitive business or operational data not intended for external audiences. Access on a need-to-know basis. |
| `internal` | General internal information. Not sensitive, but not approved for public release. Available to all staff. |
| `public` | Approved for unrestricted external sharing. No access controls required. |

### Classification guide

- **`restricted`** — Does the model contain PII, patient health data, credentials, or data covered by a privacy regulation or data-sharing agreement? → `restricted`
- **`confidential`** — Is this sensitive operational or business data that should not leave the organisation? → `confidential`
- **`internal`** — Is this general internal data with no specific sensitivity, but not meant for public release? → `internal`
- **`public`** — Has this data been explicitly approved for public release? → `public`

### Relationship to `pii`

The `pii` field flags models containing personally identifiable information columns. A model can have `pii: true` at any classification level, though in practice PII models are almost always `restricted`.
