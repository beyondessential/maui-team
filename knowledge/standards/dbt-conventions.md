# dbt Conventions

SQL formatting: [`sql-conventions.md`](sql-conventions.md). This file covers
model structure, tooling, documentation.

## Model layers

Four OMOP-mapped layers (`ref__`, `can__`, `der__`, plus the future `vocab__`),
with BES-specific layers alongside (`lkp__`) and above (`metric__`, `ds__`,
`models/reports/`). Rationale:
[`../architecture/data-architecture/decisions.md`](../architecture/data-architecture/decisions.md) D2.

```
sources / logs
     │
   bases ──┬── ref__ ──────────┐
           ├── lkp__ ──────────┤
           ├── surveys ────────┤   (generated pivot of survey responses)
           │                   ├── can__ ── der__ ── metric__ ── ds__ ── reports
           └── facts (legacy)
```

`int__` ephemerals may sit between any two layers.

| Layer | Naming | Purpose | Materialisation |
|---|---|---|---|
| `sources` / `logs` | declared in `sources.yml` | Raw tables — never modified | n/a |
| `bases/` | `base__<entity>` | **Only** layer reading `public.*` (D10). Filters deleted, normalises naming. Package-managed in `tamanu-source-dbt` | `view` (always) |
| `ref__` | `ref__<entity>` | OMOP health-system wrappers (`ref__care_site`, `ref__location`, `ref__provider`) | `view` |
| `lkp__` | `lkp__<entity>` | Curated lookups (groupings, mappings, codings). Seed-backed | `view` (always) |
| `surveys` | `<survey_id>` | Generated pivot of survey responses — one column per question per survey form. Regenerated by script; never edited manually (see repo `AGENT.md` for the script). Joined as a base by `can__measurement`, `can__observation`, and other downstream consumers | `view` |
| `can__` | `can__<entity>` | OMOP-lite clinical-event facts (`can__person`, `can__visit_occurrence`, …) | env-aware |
| `der__` | `der__<element>` | Derived constructs: cohorts (`der__cohort_<program>`), eras, episodes | env-aware |
| `metric__` | `metric__<id>` | Health indicators (shared by Tamanu reports + Tupaia). Mandatory `metric_definitions.csv` entry | env-aware |
| `int__` | `int__<description>` | Shared intermediate logic. No `order by` | `ephemeral` |
| `ds__` | `ds__<description>` | Consumer-shaped datasets. No `order by` | env-aware |
| `models/reports/` | `<description>_line_list` etc. | Tamanu reports — translations, formatting, configs. `order by` allowed | `view` |

### Legacy layers (active)

> **Sunset.** Delete once Phase 5 ([refactoring-guide.md](../runbooks/refactoring-guide.md))
> completes.

| Layer | Naming | Status |
|---|---|---|
| `facts` | `fct__<description>` | Active. No `order by` |
| `dim__` | `dim_<entity>` *(single underscore — historical, predates the double-underscore convention; do not "fix")* | Soft retirement → `ref__` / `can__` |

## Materialisations

| Type | When |
|---|---|
| `view` | Lightweight models (default) |
| `table` | Frequently queried |
| `incremental` | Large, append-only |
| `ephemeral` | Reusable logic, no materialisation |

### Environment-aware (`target.name` switch)

Models in the transitive closure of `models/reports/` ship to production as
views via the compiled bundle, while running on the replica with heavier
materialisations. Switch on `target.name`:

```sql
{{ config(
  materialized = ('view' if target.name.startswith('reporting_') else 'incremental'),
  unique_key = 'patient_id',
  incremental_strategy = 'merge',
) }}
```

- `reporting_*` → compiles the production bundle. Must be production-safe SQL
  (no Python models, valid against production schema, tractable as a view)
- `analytics_*` → runs on replica; `incremental` / `table` permitted

If a model can't be made production-safe, keep it out of any report chain — it
can live as a replica-only `metric__` consumed via Data Tables. See
[architecture](../architecture/data-architecture/production-promotion.md) and
[D9](../architecture/data-architecture/decisions.md).

`bases/`, `lkp__`, `models/reports/` are always views regardless of environment.

## File naming

- `<model_name>.sql` paired with `<model_name>.yml`
- Report configs: `models/reports/config/<model_name>.json`
- The `"query": "replace this"` placeholder in report configs is expected; the
  actual SQL is injected at compile time

## Documentation

- Mandatory `.yml` for non-report models (every layer in the tables above)
- Document all columns (≥ short description)
- Report models need their JSON config
- `metric__` models need a row in `metric_definitions.csv`
  ([D5](../architecture/data-architecture/decisions.md))
- Column descriptions are public-facing — they surface in dbt docs, data
  tables, and the reporting UI seen by end users. Describe what the column
  is, not how it's wired up. Don't reference SDD anchors (`BL-NNN`,
  `AC-NNN`), upstream base column names, or internal implementation
  details. Those belong in the spec or in `-- BL-NNN` code comments, not
  in the user-facing description.

## Testing

| Kind | File | Use for |
|---|---|---|
| Schema generic | inline `tests:` in `<model>.yml` | column invariants (`not_null`, `unique`, `relationships`, `accepted_values`) |
| Unit (dbt 1.8+) | `unit_test__<model>.yml` | scenario coverage against mocked inputs |
| Singular | `data_test__<model>.sql` | cross-model invariants |

- One singular-test file per model. Use `union all` for multiple assertions;
  tag each row with a `failed_ac` discriminator
- `bases/` need no tests (sources carry theirs)
- `ds__` need `not_null` + `unique` on PKs
- SDD-aligned tests: `ac_NNN_<model>_<assertion>` (schema) or `failed_ac`
  column (singular)
- Run `dbt test` before committing

## Code quality

- No `order by` outside `models/reports/`
- **`bases/` is the only layer reading `public.*`** (D10). Every other layer
  uses `{{ ref('base__...') }}`. See
  [D10](../architecture/data-architecture/decisions.md) for rationale
- Check downstream refs before deleting a model:
  `grep -r "ref('<model_name>')" models/`
- Never hardcode schema/table names — always `{{ ref(...) }}`
- `{{ source(...) }}` only in `bases/`

## Date/time

Use centralised vars from `dbt_project.yml` (`{{ var("date_format") }}` etc.);
never inline format strings.

## Translations

Use the project translation macro; never hardcode labels. Macro name and CSV
path are repo-specific — see repo `AGENT.md`.

## Packages

- Pin specific tags in `packages.yml`; no floating refs
- `dbt deps` after version updates
- Prefer scripts from `dbt_packages/` over local duplicates

## Do not review

`models/sources/`, `compiled/`, `models/surveys/`, auto-generated report list
files.

## Pre-commit

```bash
sqlfluff fix .
dbt test --profiles-dir config
# repo-specific validation (see AGENT.md)
```

## Repo-specific

Add layer-specific rules, variables, macros, translations, validation paths in
the repo's `AGENT.md`.

## Derived elements (`der__`) — interaction with broader conventions

Canonical pattern: [`derived-elements-conventions.md`](derived-elements-conventions.md).
This section captures the dbt-conventions touchpoints.

### Shared-pivot rule

Every consumer (Tamanu reports, Tupaia, future surfaces) sources from the
`der__` model. Never re-derive cohort membership downstream. Shared aggregation
logic → macro called from both Tamanu and Tupaia models.

### Materialisation

`der__` follows the env-aware pattern above.

### `map__<system>_<domain>` external-system mappings

`map__<system>_<domain>` is the general prefix for models and seeds that
translate Tamanu identifiers to an external system's identifiers. The
`<system>` token names the destination (`omop`, `dhis`, `tupaia`, …); the
`<domain>` names the entity domain (`sex`, `ethnicity`, `data_element`,
`org_unit`, `facility`, …).

Examples:

| Model | What it binds |
|---|---|
| `map__omop_sex` | Tamanu sex codes → OMOP concept IDs (SNOMED) |
| `map__omop_ethnicity` | Tamanu ethnicity codes → OMOP concept IDs |
| `map__dhis_org_unit` | Tamanu facility IDs → DHIS2 org-unit UIDs |
| `map__dhis_data_element` | Tamanu indicator names → DHIS2 data-element UIDs |
| `map__tupaia_facility` | Tamanu facility UUIDs → Tupaia entity codes |

**Where each lives:**

- **Universal mappings** (apply to every deployment, e.g. `map__omop_sex`,
  `map__omop_ethnicity`) → `tamanu-source-dbt`.
- **Deployment-specific mappings** (per-instance UIDs, local code maps,
  facility bindings) → `tamanu-dbt-<deployment>`.

**Materialisation:** seed-backed by default (the destination's IDs are
static reference data and benefit from CSV-style review). View-over-`(values
…)` is acceptable when the dataset is small enough that authoring a CSV adds
no value and the SQL keeps it inline-reviewable. Either way, the model is a
`view` — no incremental or table materialisation; the data is small and
slow-changing.

**OMOP concept shadow columns** (SNOMED, LOINC, RxNorm) go in `can__` and
`der__` only — never in base models. Schema for `map__omop_*`:
`local_code`, `local_name`, `concept_id`, `concept_name`, `vocabulary_id`.
Other `map__<system>_*` schemas are system-specific; document them in the
model's `.yml`.

**`map__<system>_*` ≠ `lkp__`.** `map__` translates between code systems;
`lkp__` is operational join targets (analytic groupings, hierarchies,
deployment-local codings) used by `can__`, `der__`, `metric__`, and `ds__`.
A model that holds (Tamanu facility UUID → external entity code) is
`map__<system>_facility`; a model that holds (age in years → analytic age
band) is `lkp__age_group`. Use `map__` when the value-mapping crosses
system boundaries; use `lkp__` when it stays in Tamanu/analytic space.
