# Open questions

Questions the architecture has surfaced but not answered. Each carries an
owner and a target phase. Track resolution against Linear; close items here
when the resolution is captured in a standards file or in an updated decision.

| ID | Question | Owner | Due |
|---|---|---|---|
| OQ-001 | Deprecation signalling for `data-staging` models - annotation in `.yml`, `status: deprecated` tag, separate deprecation seed? | Data Lead | Phase 1 |
| OQ-002 | Modelling pattern for non-Tamanu data in `data-warehouse` RDS - `metric__` layer settled (wide-format outputs, registry-documented definitions per D3) and dbt project home settled (`bes-data-pipelines` repo). Still open: canonical layer naming (given `can__` is the OMOP-shape canonical for clinical data; non-Tamanu sources aren't OMOP-shaped) and whether the metric registry seed lives in `tamanu-source-dbt` or `bes-data-pipelines`. Per-deployment variation likely doesn't apply - non-Tamanu sources aren't typically version-skewed per deployment. | Data Lead | Phase 1 |
| OQ-003 | Cross-source modelling - when Tupaia needs a value combining Tamanu and non-Tamanu data (e.g. drug availability per condition prevalence), where does that join happen? Options: a single Data Table querying one DB and report-server joining (light cross-source); foreign data wrappers between the two RDS instances (heavier but lets one SQL query span both); per-case Dagster jobs that materialise a combined dataset in one DB (last resort - re-introduces replication-like coupling). Defer until the single-source patterns are settled. | Data Lead + Tupaia engineering | Post-Phase 0 |
| OQ-005 | Whether and when to add a `vocab__` layer - OHDSI Athena concept tables loaded into `tamanu-source-dbt`. Deferred indefinitely; revisited only on a concrete trigger (concept hierarchy queries needed in production, full OHDSI tool compatibility required, or a research / federation partnership requesting it). Until then `map__omop_<domain>` seeds serve the lightweight role. Storage and refresh cadence for the full Athena vocabulary (~10M concepts + relationships) needs scoping when this is taken on. | Data Lead | Deferred |
| OQ-006 | Tupaia standard visual template library - repo location. Options: new repo (`tupaia-config` or similar); subdirectory in `bes-data-pipelines`; co-located with `tamanu-source-dbt`. Decision needed before any template work starts. | Data Lead | Phase 1 (early) |
| OQ-007 | Tupaia standard visual template library - template schema. Two schemas needed (dashboard items and map overlays), each matching what Tupaia's admin panel import accepts. Requires joint scoping with Tupaia engineering and likely a prototype before settling. | Data Lead + Tupaia engineering | Phase 1 |
| OQ-008 | Entity binding for map overlays - source of truth and sync for the Tamanu↔Tupaia facility mapping. The architecture answers *where* the mapping lives (`lkp__facility_tupaia_mapping` in `tamanu-dbt-<deployment>`). Still open: *who* maintains the seed (the data team via the deployment project, Tupaia engineering via the admin panel, or both with a sync) and *what* is the source of truth for Tupaia entity codes. Also: add an `entity_grain` column to `metric_definitions.csv` so templates know which Tupaia hierarchy level a metric pins to. | Data Lead + Tupaia engineering | Phase 1 |
| OQ-009 | Migration path for existing Tupaia admin-panel config to template-driven config. Multi-phase: which dashboard items and map overlays migrate first (high-traffic, well-defined), what stays in the admin panel longer-term (deployment-specific custom visuals), how the cutover is staged. Depends on OQ-006 and OQ-007 landing first. | Data Lead + Tupaia engineering | Post-Phase 0 |
| OQ-010 | Governance for AI-assisted ad-hoc reporting - which user roles can run agent-generated queries, against which data (sensitive vs non-sensitive), whether queries are reviewed before execution or whether guardrails (read-only access, query timeouts, row limits) suffice, how the agent's reasoning is audited (question, generated SQL, returned answer). | Data Lead + Tupaia engineering | Post-Phase 0 |
| OQ-011 | Integration of AI-assisted ad-hoc reporting with Tupaia's existing viz-builder for user-defined indicators - replace, complement, or operate as a separate surface. Affects how end users discover the capability and where saved Data Tables surface. | Data Lead + Tupaia engineering | Post-Phase 0 |
